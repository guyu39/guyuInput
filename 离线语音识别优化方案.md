# 离线语音识别优化方案 — go-whisper 集成

## 1. 目标

引入 go-whisper（whisper.cpp 的 Go 绑定），使应用在**无网络环境**下也能进行语音识别，纯 CPU 计算，无需 GPU。

## 2. 现状分析

### 2.1 当前 ASR 架构

```
┌─────────────────────────────────────────────────┐
│  audio.Capture (16kHz mono float32)              │
│         ↓ chan []float32                         │
│  asr.Dispatcher                                  │
│    ├── online  → asr.XunfeiClient (WebSocket)    │
│    └── offline → nil (暂未实现)                   │
│         ↓ chan RecognitionResult                 │
│  input.Dispatcher → 文本注入                       │
└─────────────────────────────────────────────────┘
```

- `asr.Engine` 接口：`Recognize(<-chan []float32) (<-chan RecognitionResult, error)`
- 离线引擎传入 `nil`，`Dispatcher` 在离线/降级场景直接报错
- 音频采集已输出 16kHz mono float32，与 Whisper 要求的格式一致

### 2.2 go-whisper 关键特性

| 项目 | 说明 |
|------|------|
| 底层 | whisper.cpp C++ 库，通过 CGo 调用 |
| 模型格式 | GGML 量化模型（`ggml-*.bin`） |
| 输入格式 | 16kHz mono float32 PCM |
| 流式输入 | **支持**。`whisper_full_with_state` C API 可增量喂入音频（Go 绑定需简单 cgo 封装） |
| 流式输出 | **支持**。Go 绑定的 `Process` 接收 `SegmentCallback`，每解码一个片段就回调一次，实时输出部分文本 |
| 线程 | 支持多线程 CPU 推理，`nthreads` 参数控制 |
| 语言 | 支持自动检测或指定（中文设为 `"zh"`） |

### 3.3 WhisperClient 设计

```go
// internal/asr/whisper.go

type WhisperClient struct {
    modelPath string   // 模型文件路径
    nthreads  int      // CPU 线程数
    language  string   // 识别语言
    mu        sync.Mutex
}

func NewWhisperClient(modelPath string) *WhisperClient

// 实现 Engine 接口
func (w *WhisperClient) Recognize(audioCh <-chan []float32) (<-chan RecognitionResult, error)
func (w *WhisperClient) Mode() ASRMode
func (w *WhisperClient) Close() error
```

**流式识别流程**：

1. 从 `audioCh` 持续读取音频块，累积到缓冲区
2. 每累积 ~500ms 音频（约 8000 样本），调用 `whisper.Process(buf, callback)`
3. `SegmentCallback` 每解码出一个文本片段就通过 `resultCh` 推送 `ResultPartial`
4. 录音结束（channel 关闭）时，最后一次调用 `Process`，推送 `ResultFinal`

```go
// 流式伪代码
func (w *WhisperClient) Recognize(audioCh <-chan []float32) (<-chan RecognitionResult, error) {
    resultCh := make(chan RecognitionResult, 10)
    go func() {
        defer close(resultCh)
        var buf []float32
        lastLen := 0
        for samples := range audioCh {
            buf = append(buf, samples...)
            // 每积累 500ms 音频做一次增量识别
            if len(buf)-lastLen >= 8000 {
                ctx.Process(buf, func(seg Segment) {
                    resultCh <- RecognitionResult{Type: ResultPartial, Text: seg.Text}
                })
                lastLen = len(buf)
            }
        }
        // 最终识别
        ctx.Process(buf, func(seg Segment) {
            resultCh <- RecognitionResult{Type: ResultFinal, Text: seg.Text}
        })
    }()
    return resultCh, nil
}
```

**两种流式策略对比**：

| 策略 | 优点 | 缺点 | 选用 |
|------|------|------|------|
| **周期性重处理**：积累音频，每 N ms 调用 `Process(累积buf)` | Go 绑定原生支持，实现简单 | 每次从头解码，短句 (<10s) 开销可忽略 | **MVP** |
| **状态式增量**：cgo 封装 `whisper_full_with_state`，只喂增量音频 | 无重复计算，效率最高 | 需要写 CGo 胶水代码 | V2 优化 |

MVP 阶段用周期性重处理即可，10 秒以内短句的额外 CPU 开销在 4 核机器上 < 200ms。

### 3.4 Dispatcher 改造点（最小修改）

`app.go` 中初始化离线引擎：

```go
// 替换原来的 nil
whisperClient := asr.NewWhisperClient(modelPath)
a.asrDisp = asr.NewDispatcher(a.xunfeiClient, whisperClient, asr.ModeAuto)
```

`Dispatcher` 的 `ModeAuto` 逻辑**无需修改**——它已正确处理"在线失败→降级离线"的流程。离线引擎实现 `Engine` 接口后即可无缝接入。

### 5.1 格式兼容性

### 5.2 增量缓冲 + 周期性推理

```go
func (w *WhisperClient) streamLoop(audioCh <-chan []float32, resultCh chan<- RecognitionResult) {
    var buf []float32
    lastProcessed := 0
    ticker := time.NewTicker(400 * time.Millisecond) // 每 400ms 推理一次

    for {
        select {
        case samples, ok := <-audioCh:
            if !ok {
                // 录音结束，最终识别
                w.finalProcess(buf, resultCh)
                return
            }
            buf = append(buf, samples...)

        case <-ticker.C:
            if len(buf)-lastProcessed < 3200 { // 至少 200ms 新音频
                continue
            }
            w.incrementalProcess(buf, resultCh)
            lastProcessed = len(buf)
        }
    }
}
```

- 5 秒录音 ≈ 80,000 个 float32 样本 ≈ 320KB 内存
- 30 秒录音 ≈ 480,000 个样本 ≈ 1.9MB 内存
- 每 400ms 推理一次，用户感知延迟 < 500ms

### 6.1 Windows 构建环境


## 7. 性能优化

### 7.1 首次加载预热

应用启动时预加载模型（避免首次录音等待）：

```go
// app.startup() 中
go func() {
    whisperClient.Preload()  // 后台加载模型到内存
}()
```

### 7.2 VAD (Voice Activity Detection)

在送入 Whisper 前进行语音端点检测，裁切静音段：
- 减少无效计算（Whisper 处理音频的时间与音频长度正相关）
- 提升准确率（排除噪音/静音干扰）
- 实现方式：短时能量检测，阈值可配置

### 7.3 线程数自适应

```go
nthreads := runtime.NumCPU() / 2  // 默认使用一半核心
if nthreads < 2 { nthreads = 2 }
```

## 10. 风险与缓解

| 风险 | 缓解 |
|------|------|
| CGo 编译在 Windows 上复杂 | 提供 MSYS2 安装脚本，或预编译 .dll 分发 |
| 模型下载依赖外网 | 提供国内 HuggingFace 镜像源，支持断点续传 |
| CPU 推理延迟（中等模型可能 > 实时） | 默认 tiny 模型，用户按需升级；VAD 减少无效计算 |
| whisper.cpp 内存泄漏 | 模型实例复用，不在每次识别时重新加载 |
