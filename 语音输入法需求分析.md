# Windows 智能输入法 — 需求分析文档

> 版本：v1.2 | 日期：2026-05-23 | 技术栈：Go + React (Wails)

---

## 一、产品定位与目标

### 1.1 产品定义

一款运行于 Windows 平台的智能输入法工具，支持**语音输入**和**键盘输入**两种模式，用户可自由切换或混合使用。语音模式通过 ASR 引擎将语音实时转文字；键盘模式提供传统拼音输入能力。文本通过 Windows IMM/TSF 框架注入到当前光标所在的任意应用程序中（如 Word、微信、浏览器、IDE 等），帮助用户以最舒适的方式进行文本输入。

### 1.2 核心目标

| 维度 | 目标 | 衡量标准 |
|------|------|----------|
| **准确度** | 中文普通话识别准确率 ≥ 95% | WER（词错误率）≤ 5% |
| **响应速度** | 从说话到文字出现 < 500ms | 端到端延迟（含网络） |
| **易用性** | 3 步内完成首次使用 | 安装 → 启动 → 说话即出字 |
| **成本** | 个人用户免费/低成本 | 本地推理零费用，云端按量计费 |

### 1.3 目标用户

| 用户群 | 场景 | 核心诉求 |
|--------|------|----------|
| **文字工作者** | 长文写作、小说创作、报告撰写 | 说话即写，降低打字疲劳 |
| **会议记录员** | 会议纪要、访谈记录 | 实时转写，不遗漏要点 |
| **编程开发者** | 写注释、文档、commit message | 解放双手，口语转书面语 |
| **肢体障碍人士** | 无法高效使用键盘 | 语音作为主要输入方式 |
| **中老年用户** | 拼音不熟练、打字慢 | 说话比打字更自然 |

## 二、用户需求分析

### 2.1 用户痛点

```
痛点 1：打字慢 — 长文本输入耗时，键盘输入速度（40-80字/分）远低于语速（150-250字/分）
痛点 2：打字累 — 长时间打字导致手腕疲劳、腱鞘炎
痛点 3：切换成本 — 现有语音输入（如微信语音）无法跨应用使用，必须复制粘贴
痛点 4：识别不准 — 方言口音、专业术语识别率低，纠错比直接打字还慢
痛点 5：隐私担忧 — 语音数据上传云端，敏感内容（合同、密码）不放心
痛点 6：离线不可用 — 网络不稳定时语音输入完全失效
```

## 三、功能需求

### 3.1 核心功能（MVP）

#### F1：语音采集与实时识别

- **触发方式**：全局快捷键（默认 `Ctrl+Alt+V`）按下即录音，松开即识别
- **识别模式**：
  - 流式识别（边说边出字，实时反馈）
  - 整句识别（说完一句话后统一出字，准确率更高）
- **输出方式**：将识别文本通过 Windows IMM（Input Method Manager）或 UI Automation 注入到当前光标位置

#### F2：文本输入注入

- **注入策略**：
  - **方案 A（推荐）**：模拟键盘输入（SendInput API）—— 兼容性最好，适用于所有应用
  - **方案 B（备选）**：剪贴板注入 —— 适合大段文本，但会覆盖剪贴板内容
- **混合策略**：短文本（< 50 字）用 SendInput，长文本（≥ 50 字）用剪贴板 + SendInput(`Ctrl+V`)

#### F3：状态指示器（Floating Bar）

- 常驻悬浮窗（可拖拽、可最小化到托盘）
- 显示状态：`待机` → `录音中` → `识别中` → `已输入`
- 音量指示器：实时反馈麦克风拾音状态
- 一键切换：在线/离线模式

#### F4：键盘输入引擎

- **输入模式**：标准拼音输入，支持全拼、简拼
- **候选词**：基于词频+上下文的智能排序，候选词数量可配置（默认 5 个）
- **翻页**：`-` / `=` 或 `[` / `]` 翻页选词
- **中英切换**：`Shift` 一键切换中/英文输入模式
- **标点配对**：中文标点自动配对（如 `"` → `""`）
- **自定义短语**：用户可配置快捷短语（如 `dizhi` → 详细地址）
- **词库管理**：
  - 系统词库：通用词汇（内置，约 30 万词条）
  - 用户词库：自动学习用户输入习惯，动态调整词频
  - 专业词库：按需导入（医学、法律、IT 等）
- **输入统计**：记录输入字数、速度、常用词，用于优化候选排序

#### F4.5：键盘 + 语音混合输入

> 整合语音与键盘输入，实现无缝切换。

- **混合模式**：用户可在输入过程中自由切换键盘和语音，无需手动切换模式
- **实现机制**：
  - 悬浮窗常驻，同时接收键盘和语音输入事件
  - 语音录制时检测键盘输入，自动暂停语音识别（避免键盘声干扰）
  - 用户可组合使用：语音口述 → 键盘精细修改 → 继续语音输入
- **智能检测**：根据输入停顿和上下文判断用户是否切换输入模式
- **无缝衔接**：键盘输入不清除语音识别的中间结果
- **视觉反馈**：悬浮窗高亮当前输入模式（语音 ● / 键盘 ■）

### 3.2 增强功能（V1.1）

#### F5：标点符号自动插入

- 基于 AI 语义理解自动添加标点（逗号、句号、问号、感叹号）
- 无需用户手动说出"逗号""句号"
- 识别语义停顿：说话停顿 > 0.8s 自动插入逗号

#### F6：中英混合识别

- 中英文无缝切换，无需手动切换模式
- 例如："打开 Chrome 浏览器访问 github.com" → 识别为 `打开Chrome浏览器访问github.com`

#### F7：语音指令

| 指令 | 动作 |
|------|------|
| "换行" | Enter |
| "删除" / "删掉" | Backspace |
| "撤销" | Ctrl+Z |
| "取消" | 清除当前识别结果 |
| "逗号" / "句号" | 插入对应标点 |



### 3.3 高级功能（V2.0）

#### F9：自定义词库

- 用户可导入专业术语表（医学、法律、IT 等）
- 人名/地名优先匹配：用户添加的姓名在识别时享有更高权重
- 格式支持：TXT（每行一词）、CSV（词 + 拼音）

#### F10：方言适配

- 支持粤语、四川话、吴语等主要方言
- 方言模型按需下载（每个约 100-200MB）

#### F11：历史记录与回溯

- 保存最近的语音输入历史（本地存储）
- 支持回听原始音频 + 查看识别结果
- 一键复制历史输入内容

#### F12：离线识别模式

- 本地部署轻量级 ASR 模型（如 Sherpa-ONNX）
- 首次使用时自动下载模型文件（约 40-80MB）
- 离线模式下识别准确率略低（≈ 90%），但保证可用性
- **优先级 P2**：作为兜底方案，MVP 阶段仅保留架构扩展点，V2.0 实装

### 5.2 技术选型：Go + React（Wails 框架）

#### 5.2.1 为什么选 Go 而非 Java？

| 维度 | Go | Java (Spring Boot / GraalVM) |
|------|-----|-------------------------------|
| **分发体积** | 单二进制 ~15MB（Wails 打包后 ~25MB） | JVM 打包 ~80MB / GraalVM Native ~40MB |
| **启动速度** | < 100ms（原生二进制） | JVM 冷启动 1-3s / GraalVM ~500ms |
| **内存占用** | 空闲 ~20MB | JVM 空闲 ~150MB（-Xmx128m） |
| **系统 API 调用** | `syscall` / `golang.org/x/sys/windows` 原生调用 | JNA/JNI 调用复杂，需额外 native 库 |
| **并发模型** | Goroutine 轻量，天然适合音频流+网络并发 | 虚拟线程(21+) / 线程池，重量级 |
| **离线模型集成** | CGo 调用 Sherpa-ONNX C API | JNI 调用繁琐，跨平台编译困难 |
| **桌面框架** | Wails（Go+React，成熟稳定） | JFX / Electron（重）/ GraalVM Native UI（生态弱） |
| **跨平台潜力** | 交叉编译一行命令 | 需分别打包 JRE |

**结论**：语音输入法是**常驻后台、低资源占用、频繁调用系统 API**的工具类应用，Go 的单二进制分发 + 低内存占用 + 原生 syscall 能力是决定性优势。Java 更适合业务逻辑复杂的企业级后端，但对桌面工具类产品而言过于重量级。

#### 5.2.2 为什么选 Wails 而非 Electron？

| 维度 | Wails (Go + React) | Electron (Node + React) |
|------|---------------------|-------------------------|
| **打包体积** | ~25MB（内嵌 WebView2） | ~80-120MB（内嵌 Chromium） |
| **内存占用** | ~40MB（复用系统 WebView） | ~150-300MB（独立 Chromium） |
| **原生 API** | Go syscall 直接调用 | 需 IPC 桥接 + native addon |
| **音频采集** | Go PortAudio 原生绑定 | 需 native addon 或 Web Audio（限制多） |
| **系统集成** | 直接调用 Win32 API | 需 electron-builder + native 插件 |

**结论**：Wails 复用系统 WebView2 运行时，不内嵌 Chromium，体积和内存都远小于 Electron。Go 后端可直接 syscall 调用 Windows API，无需 IPC 桥接。

### 5.4 关键模块实现细节

#### 5.4.1 音频采集（Go + PortAudio）

```go
// 音频采集核心逻辑（伪代码）
func startRecording() <-chan []float32 {
    portaudio.Initialize()
    stream, _ := portaudio.OpenDefaultStream(1, 0, 16000, 1024, nil) // 16kHz, 单声道
    ch := make(chan []float32, 100)
    go func() {
        buf := make([]float32, 1024)
        for {
            stream.Read()
            copy(buf, stream.Buffer)
            ch <- buf
            // 同时计算音量，通过 Wails Bind 推送给前端
            volume := calculateRMS(buf)
            runtime.EventsEmit("audio-volume", volume)
        }
    }()
    return ch
}
```

- 采样率 16kHz（语音识别标准采样率）
- 单声道（语音输入无需立体声）
- 缓冲区 1024 samples ≈ 64ms/帧（延迟可控）
- 音量数据通过 Wails Events 实时推送给 React 前端渲染

#### 5.4.2 文本注入（Go syscall）

```go
// Win32 SendInput 调用（Go 原生 syscall，无需 CGO）
func SendText(text string) error {
    // 短文本：逐字符 SendInput
    if utf8.RuneCountInString(text) < 50 {
        for _, r := range text {
            sendUnicodeChar(r)
        }
        return nil
    }
    // 长文本：剪贴板 + Ctrl+V
    setClipboardText(text)
    sendKeyCombo(VK_CONTROL, 'V')
    return nil
}

func sendUnicodeChar(r rune) {
    input := []INPUT{{
        Type: INPUT_KEYBOARD,
        Ki: KEYBDINPUT{
            WScan:  uint16(r),
            DwFlags: KEYEVENTF_UNICODE,
        },
    }, {
        Type: INPUT_KEYBOARD,
        Ki: KEYBDINPUT{
            WScan:  uint16(r),
            DwFlags: KEYEVENTF_UNICODE | KEYEVENTF_KEYUP,
        },
    }}
    SendInput(2, unsafe.Pointer(&input[0]), int32(unsafe.Sizeof(INPUT{})))
}
```

- Go 的 `syscall` 包可直接调用 Win32 API，**无需 CGO**
- `SendInput` 发送 Unicode 字符，兼容所有输入法状态
- 长文本走剪贴板方案，避免逐字符输入的闪烁和延迟

#### 5.4.3 ASR 调度器（在线/离线自动切换）

```go
type ASRDispatcher struct {
    online    *XunfeiClient   // 讯飞 WebSocket 客户端
    offline   *SherpaEngine   // Sherpa-ONNX 本地引擎
    mode      ASRMode         // auto / online / offline
    mu        sync.Mutex
}

func (d *ASRDispatcher) Recognize(audio <-chan []float32) <-chan string {
    switch d.mode {
    case ModeOnline:
        return d.online.StreamRecognize(audio)
    case ModeOffline:
        return d.offline.Recognize(audio)
    case ModeAuto:  // 默认：在线优先，失败自动降级
        ch := make(chan string, 10)
        go func() {
            result, err := d.online.StreamRecognize(audio)
            if err != nil {
                log.Warn("在线识别失败，降级到离线模式", err)
                d.mu.Lock()
                d.mode = ModeOffline  // 降级后保持离线，避免反复失败
                d.mu.Unlock()
                result = d.offline.Recognize(audio)
            }
            ch <- result
        }()
        return ch
    }
}
```

- **Auto 模式**：默认在线，网络异常时一次性降级到离线，不反复重试浪费延迟
- **恢复检测**：后台每 30s ping 云端 API，恢复后自动切回在线
- **用户可见**：前端状态栏实时显示当前模式（在线 ● / 离线 ●）

#### 5.4.5 键盘输入引擎（Go 拼音解析）

```go
// 拼音输入引擎核心结构
type PinyinEngine struct {
    trie      *Trie               // 拼音→汉字映射（前缀树）
    userDict  *FrequencyMap       // 用户词频统计
    ngram     *BigramModel        // 二元语言模型
    mu        sync.RWMutex
}

// 按键事件处理
func (e *PinyinEngine) OnKeyEvent(key rune) *CandidateResult {
    e.mu.Lock()
    defer e.mu.Unlock()

    switch {
    case key >= 'a' && key <= 'z':
        // 追加拼音字母
        e.pinyinBuf += string(key)
        syllables := segmentPinyin(e.pinyinBuf) // 音节切分
        candidates := e.trie.Search(syllables)
        // 按词频+上下文排序
        sortCandidates(candidates, e.userDict, e.ngram)
        return &CandidateResult{
            Pinyin:     e.pinyinBuf,
            Candidates: candidates[:5], // 默认展示 5 个候选
        }
    case key >= '1' && key <= '5':
        // 数字键选词
        idx := int(key - '1')
        selected := e.currentCandidates[idx]
        e.userDict.Increment(selected) // 更新词频
        e.pinyinBuf = ""
        return &CandidateResult{Committed: selected}
    case key == ' ':
        // 空格选首词
        e.commitFirstCandidate()
    }
}
```

- **拼音切分**：基于音节规则（声母+韵母）的正向最大匹配，支持全拼/简拼
- **前缀树**：Trie 存储拼音→汉字映射，O(n) 查找，约 30 万词条
- **词频学习**：用户每次选词后更新本地词频表，离线优先级调整
- **候选窗口**：独立 Frameless Window，实时跟随光标位置（通过 Win32 GetCaretPos 获取）

#### 5.4.6 输入模式调度器（语音/键盘混合）

```go
type InputModeDispatcher struct {
    voiceMode  bool          // 当前是否为语音模式
    kbdEngine  *PinyinEngine
    asrEngine  *ASRDispatcher
    textInject *TextInjector
    mu         sync.Mutex
}

func (d *InputModeDispatcher) OnKeyPress(key rune) {
    d.mu.Lock()
    defer d.mu.Unlock()

    if d.voiceMode && d.asrEngine.IsRecording() {
        // 语音录制中检测到键盘输入 → 暂停语音识别
        d.asrEngine.Pause()
        d.voiceMode = false
        runtime.EventsEmit("mode-changed", "keyboard")
    }
    // 键盘输入正常处理
    result := d.kbdEngine.OnKeyEvent(key)
    if result.Committed != "" {
        d.textInject.Send(result.Committed)
    }
    // 推送候选词到前端
    runtime.EventsEmit("candidates", result.Candidates)
}

func (d *InputModeDispatcher) OnVoiceStart() {
    d.mu.Lock()
    defer d.mu.Unlock()
    d.voiceMode = true
    d.asrEngine.StartRecording()
    runtime.EventsEmit("mode-changed", "voice")
}
```
